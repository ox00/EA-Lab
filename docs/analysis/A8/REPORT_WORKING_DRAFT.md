## 1. Problem Statement
This project studies Mario-like procedural content generation with evolutionary algorithms.

The target is not only to generate valid levels, but to generate feasible levels with controllable difficulty, structural variation, and balanced layout density.

## 3. Representation And Pipeline
Current executable chain:

`random_seed -> chromosome -> decode -> phenotype -> check_constraints -> evaluate -> evolve -> render -> artifacts`

Representation:
1. genotype: fixed-length segment ID sequence
2. phenotype: decoded tile grid

Constraint layer:
1. start/goal presence
2. standability
3. reachability
4. max jumpable gap

Objective layer:
1. `difficulty_error` (minimize)
2. `structural_diversity` (maximize)
3. `emptiness_error` (minimize)

Descriptor-only metric:
1. `emptiness`

## 4. Why Two Baselines
Two baselines are retained because they answer different questions.

### EA baseline
Role:
- simple, cheap, easy to explain

Value:
- gives a stable scalarized search baseline
- useful for checking whether the whole pipeline can converge

### NSGA-II baseline
Role:
- standard multi-objective baseline

Value:
- preserves trade-off solutions
- more aligned with the actual problem structure

## 6. Current Findings
Based on the latest baseline compare:

1. EA achieves lower `difficulty_error`
2. NSGA-II achieves higher `structural_diversity`
3. NSGA-II also shows slightly better `emptiness_error`
4. both algorithms produce feasible best candidates consistently
5. raw `emptiness` remains high for both, indicating the segment library is still structurally sparse

Current interpretation:
1. EA is better for single-region convergence
2. NSGA-II is better for preserving trade-off solution sets
3. the current content library, not only the optimizer, limits density balance

## 7. Why Hypervolume Is Added As Evaluation, Not Selection
Hypervolume is introduced as a front-quality metric in this stage.

Reason:
1. it is useful for evaluating Pareto front quality
2. it strengthens algorithm comparison evidence
3. it does not require replacing the baseline search algorithm definition

Not adopted as survivor selection yet because:
1. standard NSGA-II is a cleaner course baseline
2. hypervolume-contribution selection changes the algorithm family
3. that path is closer to SMS-EMOA than vanilla NSGA-II
