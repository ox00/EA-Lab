# A11 选题分析与课题模板（Evolution of Heuristics, EoH）

## 1. 业务背景（具体到可落地场景）
A11 的核心是让大语言模型（LLM）参与启发式算法设计：通过“生成规则 -> 运行评估 -> 反馈改写”的闭环，不断进化出更优启发式。它适用于组合优化问题中“规则工程成本高、专家经验依赖强”的场景。

典型可落地场景：

1. 物流路径与调度规则自动化设计
- 场景：VRP、车辆排班、仓储波次调度。
- 痛点：手工调规则耗时，场景变化后需反复重构。

2. 生产排程与派工规则迭代
- 场景：JSSP/FJSP、动态工单调度。
- 痛点：经典规则（SPT/EDD）稳但上限有限。

3. 通用算法研发加速平台
- 场景：算法团队快速探索新启发式。
- 痛点：创意生成慢、验证成本高、知识沉淀弱。

核心点：A11 不是直接让 LLM 解优化问题，而是让 LLM 成为“启发式设计器”，通过程序化评估闭环提升规则质量。

## 2. 业务价值（可量化）
1. 研发效率价值
- 缩短启发式原型设计周期，降低专家试错成本。
- 在同等时间内探索更多规则候选。

2. 算法质量价值
- 可发现人类未显式尝试的规则组合。
- 在特定实例分布上获得更优解或更快收敛。

3. 组织能力价值
- 沉淀“提示模板 + 评估脚本 + 回归测试”标准流程。
- 形成可复用的算法共创机制（人机协同）。

可落地量化目标示例：
- 启发式设计时间缩短 30%-60%。
- 在固定预算下解质量提升 5%-15%。
- 达到阈值解质量的计算时间下降 10%-30%。
- 可复用规则库规模持续增长（版本资产化）。

## 3. 关键难点（项目风险点）
1. 规则质量波动大
- LLM 生成内容可能不可行、低效或重复。
- 需要严格语法/语义校验与执行沙箱。

2. 评估闭环成本高
- 每轮生成都要运行算法评估，时间和算力开销累计明显。
- 若评估协议不严谨，容易“误选”规则。

3. 过拟合 benchmark 风险
- 针对训练实例迭代可能牺牲未见实例泛化能力。

4. 可重复性与治理难
- Token 采样随机性、提示词变化会影响结果稳定性。
- 需要版本管理（提示、模型、参数、种子）。

5. 合规与安全风险
- 自动生成代码可能包含低质量或潜在风险逻辑。
- 需设置代码审查与执行隔离。

## 4. 落地需要的主要资源
1. 数据与问题资源
- 优化问题实例集：训练实例 + 独立测试实例。
- 任务建议：VRP、排程、装箱等规则驱动型问题。

2. 工具与技术栈
- 平台：LLM4AD 或自建 prompt-evaluate loop。
- 运行框架：Python + 评估脚本 + 自动日志记录。
- 可选：规则语法约束、静态检查器、单元测试模板。

3. 算力与成本资源
- 主要成本：LLM API token + 评估程序运行时间。
- CPU 即可完成大部分评估；复杂任务可并行。
- 必须预先设定 token 与迭代预算上限。

4. 人力与分工（4人组建议）
- 1 人：问题建模与 benchmark 管理。
- 1 人：提示工程与生成流程设计。
- 1 人：评估闭环与回归测试。
- 1 人：实验分析、可解释性与报告打包。

5. 工程保障资源
- 规则合法性检查（语法/运行时异常处理）。
- 版本追踪：prompt、模型版本、采样参数。
- 自动回归测试：新规则对历史任务不退化。

## 5. 课题模板（可直接选）
### 模板 A11-T1：LLM 生成 VRP 启发式规则
1. 问题定义
- 输入：VRP 实例特征 + 当前规则性能反馈。
- 输出：新候选启发式（构造/邻域/接受准则）。
- 目标：提升解质量并控制求解时间。

2. 数据与基线
- 数据：标准 VRP benchmark + 未见测试集。
- 基线：人工规则、固定模板 LLM 规则、随机变异规则。

3. 评价指标
- 效果指标：平均解质量、最优值差距。
- 成本指标：time-to-target、token 成本。
- 泛化指标：未见实例性能保持率。

4. 预期产出
- 高质量规则库与适用条件总结。
- 人工规则 vs EoH 规则对比报告。

### 模板 A11-T2：EoH for Scheduling（动态派工规则）
1. 问题定义
- 输入：工单状态、机器负载、历史调度结果。
- 输出：dispatching rule（优先级函数）。
- 目标：最小化平均延期与最大延期。

2. 数据与基线
- 数据：动态排程实例集。
- 基线：SPT、EDD、ATC 等经典规则。

3. 评价指标
- 调度指标：平均 tardiness、最大 tardiness、makespan。
- 规则指标：复杂度、可解释性评分、稳定性。

4. 预期产出
- 面向动态场景的规则进化路径图。
- 不同负载强度下规则推荐表。

### 模板 A11-T3：多目标 EoH（解质量-运行时间-规则复杂度）
1. 问题定义
- 输入：候选规则集合与多目标评价函数。
- 输出：Pareto 规则集合。
- 目标：联合优化解质量、运行时间与可解释性复杂度。

2. 数据与基线
- 数据：组合优化问题实例集。
- 基线：单目标 EoH、人工规则库、多目标手工调权。

3. 评价指标
- 多目标指标：HV、IGD。
- 业务指标：部署可读性、维护成本、稳定性。

4. 预期产出
- 不同业务偏好下的规则选型策略。
- 可迁移的多目标 EoH 框架。

## 6. 课程内实施建议（按 case study 节奏）
1. 第一阶段（first report）
- 聚焦一个问题域（建议 VRP 或排程）。
- 明确闭环流程：生成 -> 校验 -> 评估 -> 筛选。
- 定义统一预算：迭代轮数、token 上限、评估次数。

2. 第二阶段（final report 前）
- 做训练/测试实例分离，验证泛化。
- 增加消融：无反馈生成、无约束生成、不同提示策略。
- 报告成本收益：性能提升 vs token/算力开销。

## 7. 风险与规避
1. 风险：生成规则不可执行或无效。
- 规避：加入语法模板与自动单测门禁。

2. 风险：性能提升依赖偶然性。
- 规避：多种子、多实例重复实验并报告置信区间。

3. 风险：过拟合训练实例。
- 规避：严格未见测试集评估，限制迭代针对性调优。

4. 风险：成本失控（token 与评估时间）。
- 规避：预设预算阈值与早停策略，按收益递减停止。

## 8. 课程适配评分（用于 Pipeline 比较）
评分标准：1-5 分，5 分最好。

1. 实施难度：3/5（流程设计复杂，但可工程化分解）
2. 数据可得性：4/5（可用公开 benchmark）
3. 算力友好度：4/5（主要是 token 与 CPU 评估成本）
4. 创新空间：5/5（LLM + 进化闭环创新度高）
5. 报告与答辩展示性：4/5（流程图、规则演化轨迹较有亮点）
6. 课程周期可控性：3/5（需严格预算管理）
7. 综合推荐度：3.9/5（创新强，执行依赖流程治理）

## 9. 结论（选题定位）
A11 是“方法新颖、可讲故事能力强”的中高潜力选题，尤其适合希望展示 LLM 与优化算法融合能力的团队。若能把闭环评估与成本控制做好，A11 可在课程中形成差异化亮点；若执行经验不足，建议先从单问题域、低预算迭代的小规模版本起步，确保结果可复现。
