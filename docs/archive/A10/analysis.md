# A10 选题分析与课题模板（Neural Architecture Search, NAS）

## 1. 业务背景（具体到可落地场景）
A10 的核心目标是自动搜索神经网络结构，以替代人工反复试错的模型设计流程。NAS 典型用于“模型精度与部署约束同时重要”的场景，如移动端推理、边缘设备实时识别、云端大规模服务优化。

典型可落地场景：

1. 边缘设备模型部署
- 场景：手机端视觉模型、IoT 设备感知模型。
- 诉求：在有限算力/内存/功耗下保持可接受精度。

2. 云端模型服务成本优化
- 场景：推荐、检测、分类在线服务。
- 诉求：在 SLA 不变的前提下降低推理成本与延迟。

3. 企业 AutoML 流水线建设
- 场景：多业务模型快速上线。
- 诉求：减少专家依赖，标准化模型搜索流程。

核心点：NAS 的价值在于“自动化架构设计 + 部署约束感知”，但计算成本与实验工程复杂度很高。

## 2. 业务价值（可量化）
1. 研发效率价值
- 减少人工调结构时间，缩短模型研发周期。
- 在同等人力下探索更多结构候选。

2. 部署成本价值
- 在精度近似条件下降低参数量、FLOPs、推理时延。
- 提升每单位算力的业务吞吐能力。

3. 平台能力价值
- 形成可复用 NAS pipeline（搜索-评估-部署）。
- 可拓展为多目标优化框架（精度、延迟、能耗）。

可落地量化目标示例：
- 在相同精度下推理延迟下降 15%-40%。
- 在同等延迟预算下 Top-1 提升 1%-3%。
- 参数量或 FLOPs 下降 20%-50%。
- 模型迭代周期缩短 20% 以上。

## 3. 关键难点（项目风险点）
1. 搜索空间巨大
- 架构组合数极大，暴力搜索不可行。
- 搜索策略与空间定义直接决定结果上限。

2. 评估成本高
- 每个候选模型完整训练代价大。
- 代理评估（早停、权重共享）会引入偏差。

3. 复现难度高
- DARTS/ENAS 类方法对实现细节与超参数敏感。
- 不同硬件与框架下结论可能变化。

4. 多目标权衡复杂
- 精度、延迟、能耗常相互冲突。
- 需要明确业务偏好才能从 Pareto 解集中选型。

5. 课程周期风险
- 若任务范围过大（如 ImageNet 全量），很容易超时。

## 4. 落地需要的主要资源
1. 数据与基准资源
- 课程级建议：CIFAR-10/100 + NAS benchmark（如 EvoXBench/NAS-Bench 类）。
- 扩展级：ImageNet 子集（仅在算力充足时）。

2. 工具与技术栈
- 框架：PyTorch。
- 工具：NASLib、EvoXBench 或简化自定义搜索框架。
- 对比方法：随机搜索、DARTS、进化 NAS。

3. 算力资源
- 推荐：至少 1 张 GPU，最好支持并行实验。
- 必需：严格预算控制（GPU hours 上限、早停、候选上限）。
- CPU 可做小规模验证，不适合作为主实验。

4. 人力与分工（4人组建议）
- 1 人：搜索空间与约束定义。
- 1 人：搜索策略实现与调参。
- 1 人：评估与部署指标测量（延迟/FLOPs/参数量）。
- 1 人：实验管理、可视化与报告复现。

5. 工程保障资源
- 统一预算协议（公平比较不同方法）。
- 自动 checkpoint 与断点续跑。
- 实验日志结构化记录（架构、指标、成本）。

## 5. 课题模板（可直接选）
### 模板 A10-T1：多目标 NAS（准确率-延迟-参数量）
1. 问题定义
- 输入：候选架构空间、延迟预算、训练数据集。
- 输出：满足预算的候选网络架构集合。
- 目标：最大化精度，最小化延迟与参数量。

2. 数据与基线
- 数据：CIFAR-10/100。
- 基线：随机搜索、DARTS、进化 NAS。

3. 评价指标
- 模型效果：Top-1/Top-5。
- 部署指标：延迟、参数量、FLOPs。
- 多目标指标：HV、Pareto 前沿覆盖度。

4. 预期产出
- Pareto 架构集合与选型建议。
- 不同预算下的最优架构策略图。

### 模板 A10-T2：低预算 NAS 策略性价比比较
1. 问题定义
- 输入：固定 GPU 小时预算。
- 输出：各 NAS 方法在预算内的最佳架构。
- 目标：最大化每 GPU 小时性能收益。

2. 数据与基线
- 数据：CIFAR-10。
- 基线：ENAS、权重共享进化、随机搜索。

3. 评价指标
- 预算效率：每 GPU 小时精度增益。
- 结果质量：最终 Top-1。
- 稳定性：跨种子方差。

4. 预期产出
- 课程场景下可行的 NAS 方法选型指南。
- 提供“预算-效果”决策曲线。

### 模板 A10-T3：约束感知 NAS（能耗/内存约束）
1. 问题定义
- 输入：设备约束（内存、功耗、推理时延）。
- 输出：满足硬约束的模型架构。
- 目标：在约束内最大化模型准确率。

2. 数据与基线
- 数据：CIFAR-10 + 设备侧测量脚本（可仿真）。
- 基线：无约束 NAS、手工轻量网络（MobileNet 类）。

3. 评价指标
- 准确率、内存占用、能耗代理指标、延迟。
- 约束满足率（硬约束违规比例）。

4. 预期产出
- 面向部署约束的可执行模型筛选流程。
- 约束放宽/收紧下的性能变化分析。

## 6. 课程内实施建议（按 case study 节奏）
1. 第一阶段（first report）
- 明确任务范围：优先 CIFAR + 单一搜索空间。
- 固定预算：如总 GPU 时长上限 + 每次实验上限。
- baseline 至少 2 个：随机搜索 + 一种 NAS 方法。

2. 第二阶段（final report 前）
- 做多随机种子重复并报告方差。
- 做消融：是否权重共享、不同搜索空间大小。
- 输出“预算敏感性分析”作为核心贡献之一。

## 7. 风险与规避
1. 风险：算力不足导致搜索不充分。
- 规避：先在 benchmark 空间验证方法，再做小规模真实搜索。

2. 风险：代理评估误差导致选错架构。
- 规避：对 Top-k 架构做完整复训确认。

3. 风险：结论不可复现。
- 规避：固定随机种子、记录硬件与软件环境、公开配置文件。

4. 风险：报告只讲精度，缺少部署价值。
- 规避：必须同时报告延迟/参数量/FLOPs。

## 8. 课程适配评分（用于 Pipeline 比较）
评分标准：1-5 分，5 分最好。

1. 实施难度：2/5（搜索与评估复杂度高）
2. 数据可得性：4/5（公开数据与基准较多）
3. 算力友好度：1/5（强依赖 GPU 预算）
4. 创新空间：5/5（多目标、约束感知、预算优化空间大）
5. 报告与答辩展示性：4/5（Pareto 与预算曲线有说服力）
6. 课程周期可控性：2/5（需严格控范围）
7. 综合推荐度：3.0/5（高潜力但资源门槛最高之一）

## 9. 结论（选题定位）
A10 是“学术前沿强、资源门槛高、工程复杂度高”的冲刺型选题。若团队拥有稳定 GPU 资源并能严格执行预算化实验管理，A10 可以产出高含金量结果；若目标是稳健交付，建议将 NAS 限定为“低预算小规模验证”，或作为其他主线（如 A1/A2）的扩展部分。
