# A5 选题分析与课题模板（Adaptive Operator Selection, AOS）

## 1. 业务背景（具体到可落地场景）
AOS 的核心问题是：在元启发式优化过程中，不同算子在不同阶段、不同实例上的有效性不同，固定算子概率会造成搜索预算浪费。AOS 通过在线/离线学习动态分配算子概率，提升求解效率与鲁棒性。

典型可落地场景如下：

1. 物流与调度优化求解器
- 场景：VRP、排产、装箱等问题的元启发式求解。
- 痛点：手工设定 mutation/local search 算子概率，跨实例迁移差。

2. 工业参数优化与工艺搜索
- 场景：多参数工艺窗口优化、组合配置优化。
- 痛点：不同工况下算子有效性变化大，固定策略不稳定。

3. 通用优化平台能力增强
- 场景：企业内部“一个求解框架服务多业务”。
- 痛点：每个业务都要人工调参与算子调度，维护成本高。

核心点：AOS 本质是“算法配置自动化”，对任何基于多算子的优化器都有复用价值。

## 2. 业务价值（可量化）
1. 求解效率价值
- 在相同算力预算下更快达到可接受解。
- 降低达到目标解质量所需迭代数。

2. 研发与运维价值
- 降低人工调参与专家依赖，缩短上线周期。
- 统一 AOS 模块后，可复用于多个问题域。

3. 稳定性价值
- 在异构实例集上提升解质量一致性，减少“某些实例失效”。

可落地的量化目标示例：
- 同预算下目标函数值提升 5%-15%。
- 达到同等解质量的时间缩短 15%-40%。
- 跨实例方差下降 10%-30%。

## 3. 关键难点（项目风险点）
1. 奖励与信用分配难
- 算子效果有延迟性，短期收益不代表长期贡献。
- 若奖励设计粗糙，易误判算子价值。

2. 探索-利用平衡难
- 过度利用会让少数算子垄断，陷入局部最优。
- 过度探索则会浪费预算，导致收敛缓慢。

3. offline-online 协同难
- 离线学到的先验在新实例上可能失效。
- 在线更新太快又可能造成策略震荡。

4. 评价口径不完整风险
- 只看最终最优值可能忽略速度、稳定性、泛化。
- 需要多维评价，实验设计复杂度高。

## 4. 落地需要的主要资源
1. 数据资源
- 训练实例集 + 独立测试实例集（规模/难度分层）。
- 至少一个公开问题库（如 QAPLIB、VRP benchmark）。

2. 工具与技术栈
- 基础求解器：GA/EA/LS 框架（可基于现有代码）。
- AOS 模块：Bandit（UCB/Thompson）、Adaptive Pursuit、RL。
- 实验工具：参数配置管理、批量运行脚本、结果统计模块。

3. 计算资源
- CPU 中等可完成课程级实验。
- 若进行大规模实例 + 多随机种子，建议多核并行。
- GPU 一般非必须。

4. 人力与分工（4人组建议）
- 1 人：算子池设计与求解器集成。
- 1 人：AOS 策略实现（bandit/RL）。
- 1 人：实验设计与统计检验。
- 1 人：可视化、报告与复现实验打包。

5. 工程保障资源
- 明确算子调用日志（何时选、贡献多少）。
- 统一随机种子与实例分层评估协议。
- 失败案例回放与 ablation（去掉某策略的消融）。

## 5. 课题模板（可直接选）
### 模板 A5-T1：面向 VRP 的 Bandit-AOS 在线算子选择
1. 问题定义
- 输入：当前解状态、算子池、历史改进记录。
- 输出：每一步应调用的算子。
- 目标：在固定迭代预算内最大化解质量提升。

2. 数据与基线
- 数据：Solomon/自建 VRP 实例集。
- 基线：固定算子概率、Adaptive Pursuit、随机选择。

3. 评价指标
- 性能指标：最终目标值、达到阈值时间（time-to-target）。
- 稳定性指标：多种子均值与方差。
- 泛化指标：未见实例上的相对改进率。

4. 预期产出
- 显示不同阶段算子偏好的动态变化。
- 给出“问题规模-算子策略”对应关系。

### 模板 A5-T2：Hybrid AOS（离线先验 + 在线纠偏）
1. 问题定义
- 输入：离线训练得到的算子先验 + 在线运行反馈。
- 输出：每轮算子概率分布更新。
- 目标：兼顾冷启动效率与跨实例适应性。

2. 数据与基线
- 数据：训练实例集（离线）+ 测试实例集（在线）。
- 基线：纯离线策略、纯在线 bandit、固定概率。

3. 评价指标
- 收敛速度、最终质量、跨实例迁移性能。
- 在线更新稳定性（震荡幅度）。

4. 预期产出
- 证明 hybrid 方案在效率与泛化之间更平衡。
- 输出可复用的 AOS pipeline。

### 模板 A5-T3：AOS 在多目标优化器中的算子自适应
1. 问题定义
- 输入：当前 Pareto 集状态、候选算子集合。
- 输出：每步算子选择与概率更新。
- 目标：同时提升 Pareto 收敛性与多样性。

2. 数据与基线
- 数据：多目标 benchmark（如课程 A1 相关测试集）。
- 基线：固定算子 NSGA-II/MOEA-D、Adaptive Pursuit。

3. 评价指标
- HV、IGD、运行时间、稳定性。
- 算子利用分布熵（衡量是否过度垄断）。

4. 预期产出
- 形成“多目标 + AOS”组合方法的可复用实验模板。

## 6. 课程内实施建议（按 case study 节奏）
1. 第一阶段（first report）
- 锁定 1 个主问题（建议 VRP 或 QAP），避免跨域过大。
- 固定 2-3 个 AOS 基线（固定概率/Adaptive Pursuit/UCB）。
- 预定义评价协议：最终质量 + time-to-target + 方差。

2. 第二阶段（final report 前）
- 完成离线训练与在线测试拆分实验。
- 做消融实验：无先验、无在线更新、不同奖励函数。
- 形成部署建议：何种实例下采用何种 AOS 策略。

## 7. 风险与规避
1. 风险：AOS 增加复杂度但收益不明显。
- 规避：先做强 baseline，并设定最低收益阈值（如 >5%）。

2. 风险：实验噪声大导致结论不稳。
- 规避：多随机种子 + 显著性检验（如 Wilcoxon）。

3. 风险：策略过拟合训练实例。
- 规避：严格训练/测试分离，报告未见实例表现。

4. 风险：可解释性不足，答辩难讲清。
- 规避：输出算子选择轨迹与阶段性贡献分析图。

## 8. 课程适配评分（用于 Pipeline 比较）
评分标准：1-5 分，5 分最好。

1. 实施难度：3/5（方法设计较抽象，实验组织要求高）
2. 数据可得性：4/5（公开 benchmark 充足）
3. 算力友好度：4/5（CPU 可做，多实例实验需并行）
4. 创新空间：5/5（奖励设计、hybrid 机制、泛化策略都可创新）
5. 报告与答辩展示性：4/5（需要用轨迹图解释，非业务图直观型）
6. 课程周期可控性：3/5（需严控实验范围与问题域）
7. 综合推荐度：3.9/5（方法创新强，交付风险中等）

## 9. 结论（选题定位）
A5 是“方法创新导向”的选题，适合希望展示算法设计与实验严谨性的团队。若团队偏向工程落地展示，A5 不如 A3 直观；若你们想在方法层面做出明确贡献（尤其是 hybrid AOS 或多目标 AOS），A5 的学术含金量较高。
