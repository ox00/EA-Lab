# A1 选题分析与课题模板（Multi-objective Optimization, MOO）

## 1. 业务背景（具体到可落地场景）
多目标优化适用于“多个 KPI 同时重要且互相冲突”的决策问题。典型业务如下：

1. 同城/城配物流调度
- 目标冲突：总配送成本 vs 准时率 vs 碳排放。
- 现实约束：车辆容量、时间窗、司机工时、道路拥堵。

2. 制造排产与车间调度
- 目标冲突：完工时间（makespan）vs 延期惩罚 vs 换线成本。
- 现实约束：机器能力、工序先后关系、设备维护窗口。

3. 云资源编排与任务调度
- 目标冲突：任务时延 vs 资源成本 vs 能耗。
- 现实约束：SLA、实例容量、突发流量。

核心点：业务实际不会接受“只优化一个指标”，而是需要一组可权衡方案（Pareto 解集）。

## 2. 业务价值（为什么值得做）
1. 决策价值高
- 给业务方提供可选方案集，不是单点答案。
- 可根据阶段目标（降本、提效、绿色）选择不同折中解。

2. 管理价值高
- 支持跨部门协同：运营看时效、财务看成本、ESG 看排放。
- 提升策略透明度，便于向管理层解释决策原因。

3. 工程价值高
- 构建“偏好可切换”的优化框架：目标权重变化时无需重构全系统。
- 未来可扩展到更多约束和目标（如公平性、稳定性）。

## 3. 关键难点（项目风险点）
1. 目标定义难
- 指标口径不一致、量纲不同，容易出现目标重复或伪冲突。

2. 约束处理难
- 真实问题中硬约束多，不可行解比例高。
- 仅靠罚函数可能不稳定，需要可行性优先策略。

3. many-objective 难
- 目标超过 3 个时，算法收敛与多样性难同时保证。

4. 评价与选解难
- 不能只看“最优值”；需看解集质量和分布（HV、IGD）。
- 最终业务仍要落地单一方案，需要后验偏好选择机制。

## 4. 落地需要的主要资源
1. 数据资源
- 学术 benchmark：课程给定 many-objective benchmark。
- 业务化数据：至少 1 套可复现实例（可匿名/仿真构造）。

2. 工具与技术栈
- 快速起步：PlatEMO（MATLAB）或 `pymoo`（Python）。
- 基线算法：NSGA-II、MOEA/D、SPEA2（至少 2 个 baseline）。

3. 计算资源
- CPU 中等即可完成课程级实验。
- 若做大规模参数搜索与多随机种子重复，建议多核并行。
- GPU 非刚需（除非与深度模型结合）。

4. 人力与分工（4人组建议）
- 1 人：问题建模与约束定义。
- 1 人：算法实现与调参。
- 1 人：实验统计、可视化（Pareto 前沿、指标表）。
- 1 人：报告与复现实验流程（README、脚本）。

5. 交付保障资源
- 统一随机种子与配置管理。
- 一键复现实验脚本。
- 结果追踪表（参数-指标-结论）。

## 5. 课题模板（可直接选）
### 模板 A1-T1：多目标 VRPTW 配送优化
1. 问题定义
- 输入：客户点、需求、时间窗、车辆容量。
- 输出：每辆车路径与服务顺序。
- 目标：最小化总里程、迟到惩罚、车辆使用数。

2. 数据与基线
- 数据：Solomon benchmark + 1 套自建配送样本。
- 基线：OR-Tools（规则法）、NSGA-II、MOEA/D。

3. 评价指标
- 学术指标：HV、IGD。
- 业务指标：准时率、单位订单成本、平均延迟。

4. 预期产出
- Pareto 前沿图 + 不同业务偏好的选解策略。

### 模板 A1-T2：云任务调度多目标优化
1. 问题定义
- 输入：任务流、资源池、SLA 限制。
- 输出：任务到实例/节点的分配与执行顺序。
- 目标：最小化平均时延、P95 时延、资源成本。

2. 数据与基线
- 数据：仿真任务日志（批任务+在线请求）。
- 基线：FCFS、加权和法、SPEA2/NSGA-II。

3. 评价指标
- 学术指标：HV、IGD。
- 业务指标：SLA 违约率、每任务成本、吞吐。

4. 预期产出
- 成本-时延折中曲线 + SLA 约束下推荐策略。

### 模板 A1-T3：制造排产多目标优化
1. 问题定义
- 输入：工单、工序、机器能力、换线时间。
- 输出：机器-工单-时间排程。
- 目标：最小化 makespan、延期罚金、换线次数。

2. 数据与基线
- 数据：经典 JSSP/FJSP benchmark + 小规模自建实例。
- 基线：遗传算法、禁忌搜索、MOEA/D。

3. 评价指标
- 学术指标：HV、IGD。
- 业务指标：延期订单占比、平均在制时长。

4. 预期产出
- 车间调度甘特图 + 不同产能场景下鲁棒性分析。

## 6. 课程内实施建议（按你们 case study 节奏）
1. 第一阶段（用于 first report）
- 确定 1 个业务场景 + 2-3 个目标 + 关键硬约束。
- 锁定数据集与 2 个 baseline。
- 明确评价指标（HV/IGD + 业务 KPI）。

2. 第二阶段（final report 前）
- 做算法改进（约束处理或算子设计）。
- 完成多随机种子重复实验。
- 给出“推荐解选择规则”和失败案例分析。

## 7. 风险与规避
1. 风险：目标太多、范围过大导致实验跑不完。
- 规避：先做 2-3 目标版本，确保可复现，再扩展。

2. 风险：只做算法分数，缺少业务解释。
- 规避：每张图都配业务含义（降本多少、准时提升多少）。

3. 风险：代码不可复现影响总成绩。
- 规避：固定环境、提供 README 和一键脚本，先做“冷启动复现测试”。

## 8. 课程适配评分（用于 Pipeline 比较）
评分标准：1-5 分，5 分最好。

1. 实施难度：4/5（中等，可用成熟多目标算法起步）
2. 数据可得性：4/5（基准数据充分，业务化数据需自构造/清洗）
3. 算力友好度：4/5（CPU 可做，多实例重复实验建议并行）
4. 创新空间：5/5（目标设计、约束处理、偏好决策均可创新）
5. 报告与答辩展示性：5/5（Pareto 前沿和 trade-off 非常直观）
6. 课程周期可控性：4/5（聚焦 2-3 目标时可稳定交付）
7. 综合推荐度：4.4/5（兼顾方法性与业务价值的高性价比选题）

## 9. 结论（选题定位）
A1 是“方法通用性强、可解释性强、可扩展性高”的均衡型选题。若你们希望在课程中同时体现算法设计能力与业务决策价值，A1 很适合作为主线；建议控制在 2-3 个核心目标，优先保证复现与对比实验完整性。
